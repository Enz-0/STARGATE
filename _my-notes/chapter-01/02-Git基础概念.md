# 02 - Git 基础概念

> 🎯 **学习目标**：理解 Git 核心概念，掌握版本控制的基本原理。

---

## 一、Git 核心概念详解

### 1.1 Repository（仓库）

**定义**：Repository（简称 Repo）是 Git 用来存储项目代码及其完整历史记录的文件夹。

**理解方式**：
- 想象成一个带有"记忆功能"的项目文件夹
- 普通文件夹只保存当前状态，而仓库保存了项目从出生到现在的所有"快照"
- 每个仓库都有一个隐藏的 `.git` 目录，存储所有历史数据

**创建方式**：
```bash
# 将普通文件夹变成 Git 仓库
git init
```

**仓库类型**：
| 类型 | 说明 | 位置 |
|:---|:---|:---|
| **本地仓库** | 存储在个人电脑上的仓库 | `C:/Users/...` |
| **远程仓库** | 存储在服务器上的仓库（如 GitHub） | `github.com/...` |

---

### 1.2 Commit（提交）

**定义**：Commit 是 Git 中保存项目某一时刻完整状态的"快照"，包含：
- 当时所有文件的完整内容
- 提交者信息（姓名、邮箱）
- 提交时间戳
- 提交说明信息（message）
- 唯一的哈希 ID（如 `a3f7d2e`）

**理解方式**：
- 像游戏中的"存档点"，可以随时回到这个状态
- 给代码拍了一张"照片"，永久保存当时的模样
- 每次提交都应该是一个逻辑上完整的小改动

**创建提交**：
```bash
# 1. 将更改添加到暂存区
git add .

# 2. 创建提交（-m 后面是提交说明）
git commit -m "修复登录页面的表单验证bug"
```

**好的提交信息规范**：
```bash
# ✅ 好的提交信息 - 描述做了什么
git commit -m "添加用户注册功能"
git commit -m "修复导航栏在移动端的显示问题"
git commit -m "更新 README 中的安装说明"

# ❌ 差的提交信息 - 太模糊
git commit -m "fix"
git commit -m "update"
git commit -m "一些修改"
```

---

### 1.3 Branch（分支）

**定义**：Branch 是代码的独立开发线，允许你在不影响主代码的情况下开发新功能。

**理解方式**：
- 像科幻电影中的"平行宇宙"
- 主分支（main/master）是稳定版本，功能分支用于开发新特性
- 分支之间可以合并（merge），将功能整合到主线

**默认分支**：
- `main` 或 `master`：项目的主分支，存放稳定的生产代码

**常用分支操作**：
```bash
# 查看所有分支
git branch

# 创建新分支
git branch feature-login

# 切换到指定分支
git switch feature-login

# 创建并立即切换（现代语法）
git switch -c feature-login

# 合并分支（将 feature-login 合并到当前分支）
git merge feature-login
```

**分支工作流程**：
```
main 分支:     A---B---C---F---G
                    ↖         ↗
feature 分支:         D---E
```

---

### 1.4 Staging Area（暂存区）

**定义**：Staging Area（也称为 Index）是准备下次提交的文件临时存放区域。

**理解方式**：
- 像机场安检前的"待检区"
- 不是所有修改都必须提交，你可以选择性地将某些更改放入暂存区
- 暂存区让你能组织好要提交的内容，形成逻辑清晰的提交

**工作流程图解**：
```
┌─────────────────┐     git add      ┌──────────────┐     git commit     ┌──────────────┐
│  Working Directory│  ───────────→  │  Staging Area │  ─────────────→  │   Repository  │
│   (工作区)        │                │   (暂存区)    │                  │   (本地仓库)   │
│                 │                │              │                  │              │
│  你编辑的文件    │                │ 准备提交的文件 │                  │ 永久保存的快照 │
│  (modified)     │                │  (staged)    │                  │  (committed) │
└─────────────────┘                └──────────────┘                  └──────────────┘
```

**暂存区操作命令**：
```bash
# 查看当前状态（哪些文件被修改、是否在暂存区）
git status

# 将指定文件添加到暂存区
git add filename.js

# 将所有修改添加到暂存区
git add .

# 取消暂存（将文件从暂存区移出，但不丢失修改）
git reset filename.js

# 清空暂存区（所有文件）
git reset
```

## 二、Git vs GitHub 区别

| 对比项 | Git | GitHub |
|:---|:---|:---|
| **本质** | 版本控制软件（工具） | 代码托管平台（服务） |
| **作用** | 在本地管理代码版本 | 在云端存储和共享代码 |
| **位置** | 安装在个人电脑上 | 运行在远程服务器上 |
| **是否必须联网** | 不需要，可离线使用 | 需要联网 |
| **关系** | 可以在本地独立使用 | 基于 Git 的云端协作平台 |
| **类比** | 像 Word 软件 | 像 OneDrive 云盘 |

**关系图示**：
```
┌─────────────────┐                      ┌─────────────────┐
│   你的电脑       │      git push        │    GitHub 服务器 │
│                 │  ─────────────────→  │                 │
│  ┌───────────┐  │                      │  ┌───────────┐  │
│  │  Git 本地  │  │      git pull        │  │  Git 远程  │  │
│  │  仓库      │  │  ←─────────────────  │  │  仓库      │  │
│  └───────────┘  │                      │  └───────────┘  │
└─────────────────┘                      └─────────────────┘
```

**一句话总结**：Git 是工具，GitHub 是使用 Git 进行协作的云端平台。没有 GitHub，Git 可以独立工作；但没有 Git，GitHub 就无法使用。

## 三、Git 工作区概念补充

Git 管理文件有三个主要区域：

| 区域 | 英文 | 说明 |
|:---|:---|:---|
| **工作区** | Working Directory | 你实际编辑文件的文件夹 |
| **暂存区** | Staging Area / Index | 准备提交的文件临时存放区 |
| **本地仓库** | Local Repository | Git 保存提交历史的地方（.git 文件夹） |
| **远程仓库** | Remote Repository | 如 GitHub 上的仓库 |

**完整工作流程**：
```
1. 编辑文件          2. 添加到暂存区       3. 提交到本地仓库      4. 推送到远程
   (Working)    →      (Staging)      →    (Local Repo)    →   (Remote)
                                              ↓
                                        git commit -m "描述"
           git add .                           ↓
                ↓                        ┌─────────────┐
           ┌─────────┐                   │  永久保存    │
           │ 暂存区   │  ─────────────→   │  的提交记录  │
           └─────────┘                   └─────────────┘
```

> 💡 **AI补充** (2026-04-07):
> 
> **暂存区(Staging Area)就像快递驿站...**
> 
> 想象你要寄快递：你先把包裹打包好，拿到**快递驿站**（这就是暂存区）暂存。驿站老板会检查包裹是否合格，你可以随时拿走或调整包裹。等你确认所有包裹都准备好了，再统一**发货**（这就是 `git commit`）。
> 
> 为什么要经过驿站？因为：
> - **可以分批处理**：你有5个文件改了，但只想先提交3个，`git add` 就是把这3个放进驿站
> - **可以反悔**：放到驿站后、发货前，随时可以拿回来（`git reset`）
> - **保证质量**：发货前最后一次检查，确保不会把错误的代码提交
> 
> **类比**：快递包裹先放到驿站(暂存区)，确认无误后再发货(提交)
>
